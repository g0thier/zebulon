# Implementation Plan: Finitions d'apparence des pages

**Branch**: `main` | **Date**: 2026-07-01 | **Spec**: [spec.md](./spec.md)

**Input**: Feature specification from `/specs/002-page-appearance-polish/spec.md`,
plus the user correction list supplied during `/speckit-plan`.

## Summary

Apply a tightly scoped visual and editorial polish pass across the existing
Zzzbre showcase pages. The implementation will keep the current Astro structure,
source Markdown model, generic page renderer, reusable cards/lists/CTA
components, and global CSS, while adding page-aware rendering rules only where
the requested corrections require a different presentation.

The plan covers:

- Homepage visual asset/link handling, benefits grid, corrected shortcut links,
  and shortcut cards.
- Product pages: cleaner feature intro, roadmap indentation/empty-check display,
  CTA heading rename, security visual card, FAQ card rendering, and blue section
  badges beside status badges.
- Tarifs, support, contact, legal, clients, PME, and about/equipe presentation
  corrections from the user list.
- Source nomenclature correction from "Incubateur Nomades" to "PГ©piniГЁre
  Nomades" across editorial and layout sources.

## Technical Context

**Language/Version**: TypeScript with Astro components; Node >=22.12.0.

**Primary Dependencies**: Astro ^7.0.4 only. No new framework, UI library, or
third-party script is planned.

**Storage**: Markdown files in `edito/` are the authoritative content source;
derived in-memory page models are built by `src/data/source-map.ts`.

**Testing**: `npm run build`; manual quickstart verification on the requested
routes; visual comparison at desktop and mobile widths. Browser verification may
use the Astro dev server with `astro dev --background`.

**Target Platform**: Static public website generated by Astro.

**Project Type**: Astro marketing/showcase website.

**Performance Goals**: Keep pages static, dependency-free, and visually stable;
the polish must not add client-side runtime work for the listed pages.

**Constraints**: Preserve source traceability to `edito/`; avoid invented
marketing copy; avoid new dependencies; keep mobile layouts free of overlap,
overflow, or unreadable text.

**Scale/Scope**: Existing homepage and routed public pages under `/produit/*`,
`/cas-d-usage/pme`, `/tarifs`, `/clients/etudes-de-cas`,
`/a-propos/equipe`, `/support`, `/contact`, and `/legal`, with generic treatment
for other pages that share section/status hero badges.
