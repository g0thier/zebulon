# Implementation Plan: Ajustements visuels mineurs des pages

**Branch**: `main` | **Date**: 2026-07-01 | **Spec**: [spec.md](./spec.md)

**Input**: Feature specification from `/specs/003-page-appearance-tweaks/spec.md`,
plus the page-by-page correction list supplied during `/speckit-plan`.

## Summary

Apply a second, tightly scoped appearance pass to the existing Zzzbre showcase
pages. The implementation will keep the current Astro structure and source-led
rendering model, while correcting specific page presentation issues on `/`,
`/produit/securite`, `/tarifs`, `/a-propos/equipe`, `/contact`, and `/legal`.

The plan covers:

- Homepage screenshot placement, BĂ©nĂ©fices heading treatment, and normal text
  list rendering for the three benefits.
- Security and pricing FAQ duplicate-heading removal, plus replacement of the
  security visual panel with the existing `src/assets/Comptes.png` screenshot.
- About/team and contact section heading extraction from cards.
- Contact address line breaks.
- Legal page title/link treatment so site and contact links match the button-like
  framed links used elsewhere.

## Technical Context

**Language/Version**: TypeScript with Astro components; Node >=22.12.0.

**Primary Dependencies**: Astro ^7.0.4 only. No new framework, UI dependency, or
third-party script is planned.

**Storage**: Authoritative content remains Markdown in `edito/`; visual
screenshots already exist in `src/assets/Capture.png` and
`src/assets/Comptes.png`.

**Testing**: `npm run build`; manual route validation through the quickstart;
desktop, tablet, and mobile visual checks. If a browser server is needed, use
`astro dev --background` per project convention.

**Target Platform**: Static public website generated by Astro.

**Project Type**: Astro marketing/showcase website.

**Performance Goals**: Keep the pages static and dependency-free; screenshot
rendering must not introduce client-side runtime work.

**Constraints**: Preserve source traceability to `edito/`; keep corrections
minor; avoid invented marketing copy; avoid new dependencies; keep long emails,
URLs, and address text readable on small screens.

**Scale/Scope**: Six requested public routes: `/`, `/produit/securite`,
`/tarifs`, `/a-propos/equipe`, `/contact`, and `/legal`. Shared components may
be touched only where needed to support these routes without regressions.
