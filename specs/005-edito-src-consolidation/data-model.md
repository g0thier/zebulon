# Data Model: Consolidation edito src

## Overview

The migration introduces a normalized editorial model inside `src/` so the site
can render pages, shared navigation, and blog article routes without reading
legacy `edito/` paths at runtime.

## Entities

### 1. Editorial Page

**Purpose**: Represents a public page rendered by the homepage route or the
catch-all editorial route.

**Fields**:

- `id`: Stable internal key for the page record.
- `slug`: Public route segment or multi-segment path.
- `section`: Top-level public section label.
- `title`: Public page title.
- `subtitle`: Optional supporting title.
- `intro`: Optional introduction shown near the hero.
- `status`: Optional badge such as `Beta`.
- `blocks`: Ordered list of content blocks.
- `ctas`: Ordered list of page-level calls to action.
- `relatedLinks`: Optional related links exposed by the page.
- `visual`: Optional hero or page visual.
- `sourceKey`: Stable migrated source identifier within `src/content/`.

**Validation Rules**:

- `slug` must remain stable for all existing public routes.
- `title` must be present.
- `blocks` must preserve source order.
- `sourceKey` must be unique.

### 2. Blog Article

**Purpose**: Represents an article detail page, initially for the Motivation
series under `/ressources/blog/motivation/[slug]`.

**Fields**:

- `id`: Stable article identifier.
- `slug`: Public article slug used by the dynamic route.
- `title`: Public article title.
- `heading`: Short label or series heading if distinct from title.
- `subtitle`: Optional article subtitle.
- `status`: Optional badge.
- `category`: Category label, initially `Motivation`.
- `series`: Optional series label.
- `questionCount`: Optional informational metric exposed by the article.
- `sections`: Ordered list of article content blocks.
- `cta`: Optional primary CTA.
- `parentCategorySlug`: Route of the category page that lists the article.
- `sourceKey`: Stable migrated source identifier within `src/content/`.

**Validation Rules**:

- `slug` must be unique within the article collection.
- `parentCategorySlug` must reference an existing editorial page.
- `sections` must preserve article section order.

### 3. Layout Content

**Purpose**: Represents shared editorial data consumed by header, footer, and
sidebar components.

**Fields**:

- `id`: Stable layout record key such as `header`, `footer`, or `sidebar`.
- `title`: Public label for the layout section when applicable.
- `groups`: Ordered navigation groups or utility sections.
- `bodyText`: Optional descriptive copy.
- `contactLinks`: Optional list of email or external contact links.
- `sourceKey`: Stable migrated source identifier within `src/content/`.

**Validation Rules**:

- Known layout keys must each exist exactly once.
- Link labels and hrefs must be non-empty.
- Group order must remain deterministic.

### 4. Route Record

**Purpose**: Connects a public path to the content entity used to render it.

**Fields**:

- `path`: Public path such as `/tarifs` or `/ressources/blog/motivation/theorie-x-y`.
- `kind`: `page` or `article`.
- `entryId`: Identifier of the page or article record.
- `section`: Top-level navigation section.
- `isStatic`: Indicates generation at build time.

**Validation Rules**:

- `path` must be unique.
- Every route record must resolve to an existing content entity.
- Article route records must be generated by the article collection.

### 5. Content Block

**Purpose**: Represents structured body content reused by pages and articles.

**Fields**:

- `kind`: `text`, `list`, `cards`, `faq`, `contact`, `quote`, or `media`.
- `heading`: Optional block heading.
- `text`: Ordered list of text paragraphs.
- `fields`: Named field list.
- `lists`: Ordered list groups.
- `cards`: Ordered card entries.
- `ctas`: Ordered CTA entries scoped to the block.
- `visual`: Optional visual asset reference.

**Validation Rules**:

- Block order must remain stable.
- `kind` must match a supported renderer branch.
- Cards and lists must preserve nested order.

## Relationships

- One **Editorial Page** has many **Content Blocks**.
- One **Blog Article** has many **Content Blocks**.
- One **Layout Content** record has many navigation groups and links.
- One **Route Record** resolves to one **Editorial Page** or one **Blog Article**.
- One category **Editorial Page** can reference many **Blog Articles**.

## State Transitions

### Legacy Source Transition

1. `Legacy only`: content exists only under `edito/`.
2. `Migrated in src`: equivalent content exists under `src/content/`.
3. `Cutover`: runtime loaders read only from `src/content/`.
4. `Legacy removed`: `edito/` is deleted and no runtime reference remains.

### Route Transition

1. `Declared`: page or article metadata exists.
2. `Mapped`: route record is generated from that metadata.
3. `Rendered`: Astro page consumes the entry.
4. `Validated`: route passes build and manual checks.

## Migration Notes

- The model intentionally separates `sourceKey` from public `slug` so internal
  organization in `src/content/` can evolve without changing URLs.
- Layout content is modeled explicitly because the current site still hard-codes
  several shared links around `edito` path assumptions.
- Blog articles are modeled separately from generic pages because they need a
  dedicated dynamic route and article-specific layout behavior.
